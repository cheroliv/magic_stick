'''

== Pourquoi Magic Stick ?

Vous etes technicien terrain, formateur ou dev nomade. A chaque intervention, vous perdez 30 minutes a configurer l'ordinateur du moment. Magic Stick elimine ce probleme : **une cle USB qui boote un environnement complet et identique sur n'importe quel PC**.

* **Formation** : chaque apprenant a la meme cle, pret a bosser des le premier jour
* **Technicien FTTH/reseau** : diagnostique reseau complet quel que soit l'ordi
* **Dev nomade** : retrouver sa stack (Docker, Java, Ollama) sur n'importe quelle machine
* **Equipe technique** : environnement standardise partage, zero temps perdu en config

'''

== Comment ca marche ?

Magic Stick utilise un systeme de **partition A/B** avec persistence :

[cols="1,2,3"]
|===
| Partition | Taille | Role

| System A
| ~8 Go
| Systeme live Xubuntu actif (boote par defaut)

| System B
| ~8 Go
| Systeme live Xubuntu de secours (inactif)

| Persistence
| Reste de la cle
| Donnees utilisateur + config logiciels
|===

=== Mise a jour atomique

1. Le script `update-system.sh` detecte la partition active (A ou B)
2. Il ecrit la nouvelle ISO sur la partition **inactive**
3. Il inverse le flag de boot GRUB
4. Au prochain demarrage : nouvelle version
5. Si ca plante ? Reboot sur l'ancienne partition — **rollback instantane**
6. La partition Persistence n'est **jamais** touchee

'''

== Demarrage rapide

=== Prerequis

* Docker (pour le build)
* Une cle USB de 32 Go minimum (64 Go recommande)
* Un systeme hote Linux

=== Build de l'ISO

[source,bash]
----
# Cloner le depot
git clone https://github.com/cheroliv/magic_stick.git
cd magic_stick

# Builder l'ISO (30-60 min)
sudo scripts/build.sh

# Verifier l'ISO generee
scripts/verify.sh
----

=== Flasher sur une cle USB

[source,bash]
----
# Identifier la cle USB
lsblk

# Flasher (ATTENTION : efface toutes les donnees du peripherique)
sudo scripts/flash.sh /dev/sdX
----

=== Mettre a jour le systeme (A/B)

[source,bash]
----
# Mettre a jour la partition inactive
sudo scripts/update-system.sh /dev/sdX

# Si le nouveau systeme ne boot pas :
# selectionner l'ancienne partition dans le menu GRUB
----

'''

== Partitionnement recommande (cle 64 Go)

|===
| Partition | Taille | Type | Flag
| 1 (System A) | 8 Go | squashfs | boot
| 2 (System B) | 8 Go | squashfs | -
| 3 (Persistence) | ~48 Go | ext4 | -
|===

La partition Persistence utilise `persistence.conf` avec `/ union` pour sauvegarder les donnees entre les sessions.

'''

== Securite

* **Aucune donnee personnelle** dans l'ISO — les cles SSH et fichiers utilisateur sont dans la partition Persistence
* **Rollback automatique** si le nouveau systeme ne boot pas
* **Donnes persistantes jamais touchees** par les mises a jour systeme
* Mises a jour atomiques : si ca casse, on revient a la version precedente en un reboot
