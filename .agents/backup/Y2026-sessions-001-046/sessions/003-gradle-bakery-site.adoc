= Session 003 — Gradle + Bakery-gradle Site Statique

**Date** : 2026-04-23 | **Statut** : OK | **Type** : feature

== Contexte

Suite de la session 002 (Git + Push GitHub). L'utilisateur a demande la mise en place d'un site de documentation statique genere par le plugin bakery-gradle et deploye sur GitHub Pages (branche gh-pages). Apres analyse du style de code des 8 repos Gradle de l'utilisateur (bakery-gradle, cheroliv, plantuml-gradle, slider-gradle, newpipe-gradle, codebase-gradle, readme-gradle, jhipster-gradle-plugins), l'agent a reproduit les memes conventions et structures.

L'utilisateur a precise : pas de CNAME, le site sera sur cheroliv.com/magic-stick. Le dossier `bake/` ne doit pas etre dans .gitignore car il est dans le layout de build deja couvert par `build/`. Le `site.yml` contient des credentials et reste dans .gitignore, injecte par la CI via le secret `BAKERY_GRADLE_PLUGIN`.

== Problemes rencontres et resolus

1. **`./gradlew: No such file or directory`** en CI — le wrapper Gradle manquait. Genere via SDKMAN localement, commit et pousse.
2. **`Cannot convert '' to File`** / `KotlinInvalidNullException: cname` — le champ `cname` etait vide dans site.yml. Le plugin bakery attend une valeur non-null. Corrige : `cname: "magic-stick"`.
3. **gh CLI non installe** — installe a partir du binaire GitHub dans `~/apps/gh`, avec script de mise a jour `gh-update.sh`.

== Lecons apprises

* Le dossier `bake/` est un artefact dans le layout de build Gradle, pas besoin de l'ajouter au .gitignore — `build/` le couvre deja
* Pas de CNAME pour un site projet GitHub Pages sous un domaine personnalise — le site est servi sous cheroliv.com/magic-stick
* Le secret `BAKERY_GRADLE_PLUGIN` contient le contenu complet du site.yml, injecte par la CI avant le build
* `cname` est non-nullable dans le plugin bakery — ne jamais le laisser vide
* Le Gradle wrapper doit etre commit dans le repo pour que la CI fonctionne
* Toujours tester en local d'abord (`./gradlew -i publishSite`) avant de deboguer la CI
* Installer les outils dans `~/apps/` en binaire, jamais via apt — fournir un script de mise a jour dans le meme dossier
* Le token `gho_` (fine-grained) fonctionne pour l'API GitHub, le `ghp_` (classic) est expire
