== Regles Absolues

=== 0. COMMANDES DESTRUCTRICES — INTERDICTION FORMELLE

_L'agent NE DOIT JAMAIS_ executer de commandes destructrices SANS permission explicite de l'utilisateur.

* Interdit : `dd`, `mkfs`, `mkfs.ext4`, `rm -rf` (hors repertoire de build), `parted`, `fdisk`, toute commande ecrivant sur `/dev/sdX`
* Autorise : `bash -n` (verification syntaxe), `lsblk`, `ls`, `cat`, scripts de build (avec permission)

=== 1. COMMITS/GIT

_L'agent NE DOIT JAMAIS_ executer de commit, push, merge, ou toute commande git modifiant l'historique _SANS permission explicite de l'utilisateur_.

* Autorise : `git status`, `git diff`, `git log`, `git show` (lecture seule)
* Interdit : `git add`, `git commit`, `git push`, `git merge`, `git rebase` (sauf ordre explicite)

_Procedure obligatoire avant tout commit_ :
. Montrer les modifications (`git diff --stat`)
. Demander : "Veux-tu que je commit ces changements ?"
. _Attendre confirmation explicite_ ("oui", "commit", "vas-y")
. _Seulement apres_ : executer le commit

=== 1b. FICHIERS DE CONFIGURATION — REGLE ABSOLUE DE SURETE

**PRINCIPE** : Tout fichier de configuration (YAML, JSON, XML, TOML, properties, .env) peut contenir des donnees sensibles non versionnees. Ne **jamais** mettre en danger l'integrite du travail du developpeur.

**INTERDICTIONS** :

* **NE JAMAIS ecraser** un fichier config avec un `Write` complet quand un `Edit` partiel suffit
* **NE JAMAIS remplacer** des valeurs sensibles (tokens, mots de passe, cles API, URLs prod) par des valeurs factices
* **NE JAMAIS modifier** sans verifier `git check-ignore` et `git ls-files` — un fichier non tracke n'a pas de `git restore`

**Procedure OBLIGATOIRE** avant modification :

. `git check-ignore <fichier>` + `git ls-files -- <fichier>`
. Si non tracke : **CREER UNE SAUVEGARDE** (`cp <fichier> <fichier>.bak`)
. **PRESERVER** les valeurs sensibles intactes
. Privilegier l'**edit partiel** sur l'ecriture complete
. **Verifier** apres modification que les valeurs sensibles sont toujours la

_Lecon Session 2 (bakery-plugin)_ : Ecraser un site.yml non versionne a detruit des tokens reels. Sans sauvegarde git (fichier ignore), restauration impossible sans fallback externe. Le developpeur aurait ete bloque definitivement.

=== 2. BUILD ISO — PERMISSION REQUISE

_L'agent NE DOIT JAMAIS_ lancer de build ISO (`sudo scripts/build.sh`) ou de flash (`sudo scripts/flash.sh`) _SANS permission explicite de l'utilisateur_.

* Autorise : Modifier les scripts, les configs, les package-lists
* Interdit : Executer le build ou le flash sans confirmation

_Raison_ : Le build ISO prend 30-60 minutes. Le flash ecrit sur une cle USB reelle.

=== 3. PROCEDURE DE FIN DE SESSION — EXECUTION COMPLETE OBLIGATOIRE

_REGLE ABSOLUE_ : Quand l'utilisateur declenche la fin de session, l'agent DOIT executer _TOUTES les etapes_ de la procedure _SANS EXCEPTION et SANS OMISSION_.

Les 6 etapes OBLIGATOIRES (dans l'ordre) :

. _Creer l'archive_ `.agents/sessions/{N}-{titre}.adoc`
. _Mettre a jour PROMPT_REPRISE.adoc_ pour la session N+1
. _Mettre a jour SESSIONS_HISTORY.adoc_ avec la session N
. _Mettre a jour INDEX.adoc_ (session en cours, roadmap, fichiers)
. _Mettre a jour SCRIPT_VERIFICATION.adoc_ si des scripts ont ete ajoutes/modifies
. _Mettre a jour COMPLETED_TASKS_ARCHIVE.adoc_ `.agents/archives/COMPLETED_TASKS_ARCHIVE_{mois}.adoc` avec les taches realisees

_Confirmation finale OBLIGATOIRE_ : Apres avoir execute les 6 etapes, l'agent DOIT afficher :

[source]

_Aucune etape ne peut etre marquee [OK] si le fichier n'a pas ete effectivement modifie et verifie._

=== 4. OUTILS SYSTEME — INTERDICTION D'APT ET DE SUDO SUR HOTE

_L'agent NE DOIT JAMAIS_ utiliser `apt`, `apt-get`, `sudo apt install` ou toute commande modifiant les paquets systeme de l'hote.

**Installation d'outils** : Toujours installer dans `~/apps/` en binaire, jamais via apt. Fournir un script de mise a jour dans le meme dossier.

**Operations Docker** : Toutes les operations necessitant des droits superieurs (montage ISO, build, etc.) se font **dans le conteneur Docker**, jamais avec `sudo` sur l'hote.

_Lecon Session 3_ : L'agent a tente `sudo apt install gh` ce qui a pourri les depots apt. Depuis, les outils s'installent en binaire dans `~/apps/` avec un script `xxx-update.sh`.

_Lecon Session 012_ : L'agent a tente `sudo mount` sur l'hote alors que toute operation privilegiable se fait dans le conteneur Docker.

=== 5. FICHIERS PERSONNELS — INTERDICTION DE MODIFIER

_L'agent NE DOIT JAMAIS_ modifier, ecraser ou supprimer un fichier personnel de l'utilisateur (`~/.gitconfig`, `~/.bashrc`, `~/.ssh/config`, `~/.zshrc`, etc.) _SANS permission explicite de l'utilisateur_.

**PRINCIPE** : Les fichiers `~/.*` appartiennent a l'utilisateur. Leur modifier peut casser le workflow, les credentials, ou les preferences de l'environnement. L'environnement de travail de l'utilisateur est en dehors du scope du projet.

**INTERDICTIONS** :

* **NE JAMAIS** utiliser `Write` ou `Edit` sur un fichier `~/.*` ou `~/*` qui n'est pas dans le repertoire de travail du projet
* **NE JAMAIS** executer de commande touchant `~/.gitconfig`, `~/.bashrc`, `~/.ssh/*`, `~/.zshrc`, etc. meme avec l'intention de "reparer" ou "configurer"
* **Toujours rester dans le working directory** (`magic_stick/` ou le repertoire projet actuel)

_Lecon Session 033_ : L'agent a tente de modifier `/home/cheroliv/.gitconfig` pour changer le `user.name` de `opencode` vers `cheroliv`. Sans permission explicite. Modifier les configs utilisateur hors projet est un comportrement dangereux qui peut corrompre l'environnement de travail.

_Lecon Session 041_ : Apres extraction d'un tar avec `--strip-components`, le binaire final ne se trouve pas forcement a la racine du dossier cible. Il faut toujours verifier le path reel avant de creer un symlink. Le hook affichait "installed" mais le symlink `/usr/local/bin/jetbrains-toolbox` etait mort car le binaire reel etait dans `bin/jetbrains-toolbox`. Le test ISO squashfs a mis 22 minutes a reveler le bug.
